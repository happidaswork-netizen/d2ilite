## D2I Lite 暂缓实现的迭代清单

更新时间：2026-03-09

适用仓库：`d2ilite`

### 1. 说明

当前决定先暂停 `d2ilite` 的继续开发，不在这一轮直接开做新功能。

为了避免后续接手时重新讨论一遍，这里把已经明确的方向记录下来，作为下一轮恢复开发时的优先级参考。

补充决策（2026-03-09）：

1. 后续恢复开发时，默认按“旧版冻结，新版独立重构”执行。
2. 这里记录的角色元数据需求，默认应优先在新版主工程中实现。
3. 不再要求 Tk 与新版长期双轨同步演进。

### 2. 当前已具备的基础

1. 图片元数据中已经支持写入 `role_aliases`。
2. Tk 界面里已经有“扮演角色”入口。
3. `TITI` 的智能图生图已经能消费这类元数据，并支持按文件名、原角色名、扮演角色名进行匹配。

### 3. 下一轮优先级

#### P1：把“扮演角色”从文本框升级为结构化编辑

当前问题：

1. 现在的“扮演角色”本质仍是逗号分隔文本，适合临时录入，不适合长期维护。
2. 后续如果要支持同角色多候选、轮换、配额、权重，纯文本很快会失控。

当前进展：

1. `desktop-next` 已完成单图结构化编辑原型，可维护原角色名、扮演角色名、备注、启用状态。
2. `desktop-next` 已完成目录级角色工作流：原角色名 / 扮演角色名筛选、勾选集维护、批量设置 / 追加 / 替换 / 清空。
3. 还没完成的是把这套规则继续沉到长期 `domain` 层，并扩展更复杂的角色策略字段。

建议目标：

1. 改成结构化列表编辑器。
2. 每条记录至少包含：
   - 扮演角色名
   - 备注
   - 启用/停用
3. 预留后续字段：
   - 权重
   - 次数上限
   - 优先级

#### P2：把角色元数据能力补到 `desktop-next`

当前问题：

1. 角色相关入口现在主要还在 Tk 界面。
2. 新前端虽然已经有角色编辑和目录工作流，但长期结构仍主要挂在 `workspace + features/metadata/model.ts`。

建议目标：

1. 在 `desktop-next` 中补齐：
   - 原角色名
   - 扮演角色名
   - 结构化预览
2. 保证新旧界面的数据语义一致。

#### P3：补批量角色编辑能力

当前问题：

1. 当前基础批量能力已经有了，但还缺更高阶的批量规则、反馈和回归固化。
2. 后续选角测试仍会频繁需要整组修正角色信息。

建议目标：

1. 在现有目录工作流基础上，继续扩展更高阶批量规则与反馈。
2. 补更稳定的目录索引 / 缓存与批量回归。
3. 支持后续更复杂的角色策略字段。

#### P4：统一术语

当前建议固定为三套概念，不再混用：

1. 文件名
2. 图片原角色名
   - 对应 `d2i_profile.name`
3. 图片扮演角色名
   - 对应 `role_aliases.name`

目标：

1. `d2ilite` 界面这样显示。
2. `TITI` 界面这样显示。
3. 文档、字段说明、筛选条件也统一用这套叫法。

#### P5：整理仓库后再继续大迭代

当前仓库里混有以下类型的改动：

1. Tk 元数据相关修改
2. `desktop-next` / Tauri 现代化改动
3. 启动脚本与桥接脚本
4. 本地运行产物与未跟踪目录

建议：

1. 下一轮继续开发前，先拆干净提交。
2. 至少把“角色元数据迭代”和“桌面现代化迭代”分开管理。
3. `data/` 等本地运行目录不进入版本库。

### 4. 当前不建议立刻做的事情

1. 不建议现在就继续扩写复杂选角策略。
2. 不建议在仓库仍然混杂大量本地改动时继续堆新功能。
3. 不建议同时重做 Tk 和 `desktop-next` 两套界面的角色编辑体验。

### 5. 恢复开发时的推荐顺序

1. 先做仓库整理。
2. 再收敛 `desktop-next` 的 `domain / infrastructure` 边界。
3. 然后补更高阶批量能力与角色策略字段。
4. 最后再评估是否还需要回写 Tk 侧入口。

### 6. 本轮结论

本轮先记录方向，不继续实现。

下一次恢复 `d2ilite` 开发时，优先围绕“角色元数据结构化”和“批量选角维护”展开，而不是再新开一条独立流程。
