# 当前图片动作线接手说明（2026-06-26）

## 1. 背景

D2I Lite 当前已经承担“本地图片素材查看、元数据查看/编辑、下载修复、旧版 D2I 批量下载器、公共抓取入库”等职责。接下来新增的“加名字白条”和“单图 AI 随手处理”不应再作为独立脚本散落维护，而应纳入 D2I Lite 的当前图片动作线。

这条线的目标是：让 D2I Lite 成为本地图片素材的单张处理工作台。

## 2. 产品分工

| 工具 | 职责 | 不负责 |
| --- | --- | --- |
| D2I Lite | 单张图片查看、元数据维护、下载修复、导出衍生图、单图 AI 处理、右键/Quicker 随手动作 | 大规模提示词编排、批量生成工作流 |
| TITI | 批量生成、批量编排、素材消费、创意工作台 | 单张图片日常维护入口 |
| Quicker / Windows 右键 | 入口层：把当前选中的文件路径和动作名传给 D2I Lite | 图片处理、元数据解析、窗口选中项猜测 |

核心原则：Quicker 只传参数，不承载业务逻辑；D2I Lite 负责所有图片处理与配置。

## 3. 当前仓库事实

- 入口程序：`app.py`
- 主窗口类：`app.py` 中的 `D2ILiteApp`
- 现有顶部按钮区域：`D2ILiteApp._build_ui()` 内的 `action_row`
- 现有编辑页按钮：`D2ILiteApp._build_edit_tab()`，已有 `AI自动补全`、`AI自动小传`、`下载修复` 等单图相关能力
- 元数据读取结构：`metadata_manager.ImageMetadataInfo`
- 元数据写入：`metadata_writer.py`
- Windows 打开方式注册：`register_windows_file_assoc.ps1`
- 旧版批量下载器：`legacy_downloader_gui.py`
- 队列能力：`queue_manager.py`

建议新增动作系统，不继续把所有新能力堆进 `app.py`。

## 4. 建议新增模块结构

```text
d2ilite/
  actions/
    __init__.py
    base.py                 # ImageActionContext / ImageActionResult / registry
    name_bar.py             # 加名字白条
    ai_repair.py            # 单图 AI 修复
    ai_generate.py          # 单图 AI 生图
    metadata_fill.py        # 复用现有 AI 元数据补全能力的动作封装
  services/
    image_rendering.py      # 通用图片渲染/输出命名/格式处理
    model_gateway.py        # 大模型/图像模型统一端口
  docs/
    当前图片动作线接手说明_2026-06-26.md
    加名字白条功能规格_2026-06-26.md
    AI单图动作接口规划_2026-06-26.md
```

如果短期不想大改目录，也可以先只加：

```text
d2ilite/
  image_actions.py
  image_name_bar.py
  model_gateway.py
```

但长期建议拆出 `actions/`，避免 `app.py` 继续膨胀。

## 5. 动作接口草案

```python
@dataclass
class ImageActionContext:
    image_path: str
    metadata: ImageMetadataInfo | None
    app_config: dict
    ui_parent: Any | None = None

@dataclass
class ImageActionResult:
    ok: bool
    message: str
    output_path: str | None = None
    reveal_path: str | None = None
    metadata_updates: dict | None = None

class ImageAction(Protocol):
    id: str
    label: str
    def run(self, context: ImageActionContext, **options) -> ImageActionResult: ...
```

动作系统必须同时支持 GUI 和命令行调用。GUI 按钮、右键菜单、Quicker 都应走同一个动作函数。

## 6. 命令行入口建议

D2I Lite 当前已经支持命令行传入图片路径：

```powershell
python app.py "D:\images\sample.jpg"
```

建议扩展为：

```powershell
python app.py --action name-bar "D:\images\sample.jpg"
python app.py --action ai-repair "D:\images\sample.jpg"
python app.py --action ai-generate "D:\images\sample.jpg"
```

打包后对应：

```powershell
D2ILite.exe --action name-bar "D:\images\sample.jpg"
```

Quicker 和 Windows 右键只调用这个命令，不再自行判断资源管理器当前选中项。

## 7. GUI 入口建议

短期在现有顶部按钮区域增加一个“当前图片工具”按钮，点击弹出小菜单：

- 加名字白条
- AI 修复图片
- AI 单图生图
- AI 补全元数据
- 打开动作设置

后续可改成右侧工具面板或独立 `当前图片动作` 标签页。

## 8. 配置边界

动作配置应该存在 D2I Lite 自己的配置里，而不是 Quicker 脚本里。建议配置项包括：

```json
{
  "image_actions": {
    "name_bar": {
      "output_format": "png",
      "jpg_quality": 100,
      "output_name_mode": "label",
      "bar_height_mode": "auto",
      "align": "center"
    },
    "model_gateway": {
      "provider": "openai-compatible",
      "base_url": "",
      "api_key_env": "D2I_MODEL_API_KEY"
    }
  }
}
```

不要把 API Key 明文写进仓库；本地配置或环境变量处理。

## 9. 实施顺序建议

1. 先做动作系统最小骨架：`actions/base.py` + CLI `--action` 分发。
2. 接入 `name-bar`，只依赖 Pillow，不接大模型。
3. 在 GUI 当前图片区域放入口，并让命令行和 GUI 共用同一实现。
4. 加 Windows/Quicker 调用说明。
5. 再做 `model_gateway.py`，先支持一个 OpenAI-compatible 配置。
6. 做 `ai-repair` 和 `ai-generate`，保持单图、手动确认、输出到派生文件，不覆盖原图。
7. 最后再考虑和 TITI 的批量能力共享配置或任务格式。

## 10. 验收口径

- 从 GUI 当前图片点击“加名字白条”，能生成派生图并定位打开。
- 从命令行 `--action name-bar` 调用同一张图，输出一致。
- Quicker 通过路径参数调用，不出现选错图片。
- 默认输出 PNG，避免新增 JPG 有损压缩。
- JPG 输出可选，支持质量设置，并在文档中说明 JPG 一定会重新编码。
- 输出文件命名支持直接用角色名命名。
- 原图不被覆盖，除非某个动作明确经过二次确认。
